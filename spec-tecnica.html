01

Visão geral e princípios

O sistema reage a dois tipos de gatilho: eventos emitidos pela plataforma quando algo acontece, e timers que disparam quando algo deveria ter acontecido mas não aconteceu dentro do SLA configurado. Ambos resultam em mensagens enviadas via Slack (time interno) ou WhatsApp (cliente final), através de templates configuráveis.

Toda a configuração eventos, canais, templates, SLAs vive numa matriz editável pelo time de produto sem necessidade de deploy. Engenharia entrega o motor; produto opera as regras.

Princípios de design

  • ·Event-driven. A plataforma emite eventos no event bus; o sistema reage. Sem polling.
  • ·Configurável sem deploy. Toda regra (evento → canal → template → SLA) vive em tabela editável.
  • ·Idempotente. O mesmo evento não pode disparar a mesma notificação duas vezes.
  • ·Auditável. Toda mensagem enviada vai pro log com payload, canal, timestamp e custo.
  • ·Cancelável. Todo follow-up agendado tem um evento de cancelamento atrelado. Resolveu antes? Cancela.
  • ·Respeitoso. Janela horária 8h–21h. Sem flood. Escalação só após N tentativas.

Stack proposta

Node.js / TypeScript PostgreSQL Redis (queues + locks) BullMQ (scheduler) Slack Bolt SDK WhatsApp Business Cloud API a confirmar com tech lead

A stack acima é referência; ajustar conforme padrões já estabelecidos no monorepo backend.

02

Arquitetura

O sistema é composto por seis componentes principais. A plataforma de operações é produtora de eventos; os demais módulos pertencem ao serviço de mensageria e podem ser entregues como um único serviço ou separados conforme decisão do time.

Responsabilidades

ComponenteResponsabilidade
PlataformaEmite eventos no event bus em cada ação relevante (criar operação, preencher formulário, etc).
Event BusTransporta eventos com retenção mínima de 7 dias para replay. Garante entrega at-least-once.
Event ListenerConsome o bus em consumer group próprio. Aciona engine de notificação e engine de SLA.
Notification EngineConsulta a matriz, resolve template com variáveis, valida regras (idempotência, horário), chama adapter.
SLA EngineAgenda jobs com vencimento futuro, verifica jobs vencidos a cada minuto, dispara follow-ups, cancela quando o evento de saneamento acontece.
MatrizTabelas em PostgreSQL contendo as regras de notificação e follow-up. Editável via painel admin.
Slack AdapterEncapsula chamadas à API do Slack. Lida com rate limit e retry exponencial.
WhatsApp AdapterEncapsula chamadas à WhatsApp Business Cloud API. Valida janela 24h, seleciona template HSM aprovado.

Todos os payloads compartilham os campos base abaixo, e a plataforma é responsável por emiti-los sincronamente nos pontos do código indicados na coluna "disparado quando".

  "event_name": "formulario_dados_preenchido",
  "emitted_at": "2026-04-09T14:30:12Z",
  "operation_id": "op_19432",
  "actor": { "type": "user", "id": "u_8821" }, // quem causou o evento

Política de escalação

Quando um follow-up dispara e o evento de cancelamento ainda não chegou, o sistema reagenda uma nova tentativa. Cada follow-up tem max_attempts (padrão 3). Após esgotar tentativas, o caso vai para a regra de escalação (escalation_rule_id), que tipicamente notifica o gestor da operação ou cria um ticket de atendimento manual.

Intervalo entre tentativas (sugestão padrão):

  • · Tentativa 1: no SLA original.
  • · Tentativa 2: 24h depois.
  • · Tentativa 3: 72h depois.
  • · Após 3: escala (cria ticket + notifica gestor).

08

Templates de mensagem

Copy completa, preview visual e variáveis disponíveis de cada um dos 19 templates, organizados pela etapa da operação. Notificações à esquerda (em resposta a evento), follow-ups à direita (disparados por SLA estourado). Templates Slack usam Block Kit com botões interativos; templates WhatsApp são HSM UTILITY e devem ser submetidos à Meta antes do go-live.

Convenções de copy

  • ·Slack: tom direto e operacional. Primeira linha é o título do evento; corpo entrega o contexto que o time precisa pra agir. Sempre inclui CTA primário e secundário em botão.
  • ·WhatsApp: tom caloroso mas profissional. Sempre se identifica como "Hubfi" (intermediadora) para evitar confusão com o banco. Texto na primeira pessoa, sempre com link claro e quick reply pra falar com consultor.
  • ·Variáveis: em {snake_case}, substituídas em runtime pelo Notification Engine a partir do payload do evento e dos dados da operação.
  • ·Bold no Slack/WhatsApp: usa *texto* conforme convenção dos canais o adapter converte para o formato apropriado.

Regras críticas

Esta seção lista regras que não podem ser violadas pela implementação. Cada uma tem rationale, comportamento esperado, e estratégia de teste.

Crítico

R1 · Idempotência por evento

O mesmo event_id (ULID) entrando no listener mais de uma vez (replay, retry do bus) nunca deve gerar notificações duplicadas. Implementar via tabela de deduplicação consultada antes de qualquer side-effect.

Teste: emitir o mesmo evento duas vezes e validar que notification_log só tem uma entrada por regra aplicável.

Crítico

R2 · Janela horária para o cliente

Mensagens para o cliente (WhatsApp) só podem ser enviadas entre 08:00 e 21:00 no fuso horário do cliente. Follow-ups que vencem fora desta janela são reagendados para o próximo horário permitido. Notificações Slack para o time interno seguem outra regra (24/7 OK).

Teste: simular SLA vencendo às 23h; validar que scheduled_for é atualizado para 08:00 do dia seguinte.

Crítico

R3 · Cancelamento automático de follow-ups

Antes de disparar um follow-up vencido, o pipeline obriga uma verificação em tempo real do estado da operação (não apenas inspeção do log). Caso o evento de cancelamento tenha sido emitido (mesmo que ainda não processado), o follow-up é cancelado. Pior cenário a evitar: cobrar cliente por algo que ele já resolveu.

Teste: agendar follow-up de 60s, emitir evento de cancelamento aos 59s, validar que mensagem não é enviada.

Importante

R4 · Rate limit por destinatário

Um cliente não pode receber mais de 3 mensagens WhatsApp em 24h. Um canal Slack não recebe mais de 10 mensagens em 5 min. Acima desses limites, agrupar em uma mensagem consolidada com link "ver detalhes". Configurável por destinatário no painel admin.

Teste: simular 5 follow-ups consecutivos para o mesmo cliente; validar agrupamento.

Importante

R5 · Escalação após N tentativas

Após max_attempts tentativas de follow-up sem sucesso, criar ticket manual e notificar o gestor da operação. Não tentar mais via WhatsApp/Slack pelo mesmo canal automatizado. Default: 3 tentativas.

Teste: validar criação de ticket no terceiro retry vencido sem cancelamento.

Importante

R6 · Opt-in obrigatório (WhatsApp)

Adapter WhatsApp consulta o registro de opt-in antes de enviar. Sem opt-in registrado → delivery_status: failed com motivo claro. Opt-in é capturado na criação da operação e exposto em endpoint próprio.

Teste: tentar enviar para cliente sem opt-in; validar bloqueio.

Padrão

R7 · Auditoria imutável de mudanças na matriz

Toda mudança em notification_rules ou followup_rules gera entrada em audit_log com snapshot before/after. Permite rollback e investigação de incidentes.

Roadmap de entrega

Sugestão de divisão em 5 semanas, partindo do zero. As fases podem ser paralelizadas se houver mais de uma frente. Ao final da fase 5 o sistema está pronto para go-live com as etapas 01–05.

1 Sem 1–2

Event bus + matriz

  • · Setup do event bus (Redis Streams ou Kafka).
  • · Tabelas events_catalog, notification_rules, followup_rules, audit_log.
  • · API admin de CRUD da matriz.
  • · Instrumentação inicial da plataforma para os eventos das etapas 01–02.

Saída: eventos circulando, regras editáveis.

2 Sem 2–3

SLA engine

  • · Scheduler com worker rodando a cada minuto (BullMQ ou equivalente).
  • · Tabela followup_jobs + índices.
  • · Lógica de agendamento, cancelamento e disparo.
  • · Regra de horário e escalação.
  • · Suite de testes cobrindo R1–R3.

Saída: follow-ups funcionando end-to-end com dados de teste.

3 Sem 3–4

Adapters de canal

  • · Slack Bolt app + scopes + bot user.
  • · WhatsApp Cloud API + submissão de templates HSM à Meta.
  • · Webhooks de status (ambos os canais).
  • · Retry exponencial e rate limit.
  • · Tabela notification_log + integração.

Saída: mensagens reais chegando nos canais reais.

4 Sem 4

Painel admin + observabilidade

  • · UI do admin para editar matriz com preview de mensagem.
  • · Histórico por operação (linha do tempo das mensagens enviadas).
  • · Dashboard com métricas operacionais.
  • · Alertas no monitoring.

Saída: operável e observável.

5 Sem 5

Hardening + go-live

  • · Testes de carga (1000 eventos/min simulados).
  • · Disaster recovery: replay do event bus.
  • · Documentação de runbook on-call.
  • · Rollout gradual: 10% das operações → 50% → 100%.

Saída: sistema em produção com 100% do tráfego.

Etapas 06–11 (entrega posterior)

Após o go-live com as primeiras 5 etapas, as etapas Avaliação do imóvel, Análise jurídica, Elaboração do contrato, Assinatura, Registro e Liberação do recurso entram em iterações de uma sprint cada. O motor não muda; apenas adicionamos eventos ao catálogo e regras na matriz.
